# 02 · 專案總覽

這一章讓你「鳥瞰」整個專案:它在做什麼、由哪些部分組成、資料怎麼流動。
讀完你會有一張「地圖」,之後看細節才不會迷路。

---

## 1. BandSheet 是什麼

一個給樂團用的網站,核心是「**多人即時共編和弦歌詞譜**」。主要功能:

- 註冊/登入
- 建立**樂團(band)**、邀請成員(成員有角色:OWNER 團長 / EDITOR 編輯 / VIEWER 檢視)
- 建立**歌曲(song)**,用 ChordPro 格式編輯(和弦標在歌詞上方)
- **即時共編**:兩個人同時開同一首歌,一方改、另一方 1 秒內看到
- **移調**:一鍵把整首歌升/降半音
- **版本歷史**:每次存檔留一版,可還原
- **我的歌曲**:個人歌曲庫,可分享到一個或多個樂團
- **我的最愛**、**分類(tag)搜尋**、**我的歌單(setlist)**

## 2. 領域概念(把它想成真實世界)

```
使用者 User ──屬於多個──> 樂團 Band ──有多位──> 成員(角色)
   │
   ├─建立─> 歌曲 Song ──可分享到──> 多個樂團(song_bands)
   │            └─有多個─> 版本 SongVersion
   │            └─被加入─> 歌單 Playlist(有序)
   └─可收藏─> 我的最愛(song_favorites)
```
- 歌曲「屬於建立者」,是否給樂團看,靠「分享」決定。
- 歌單是「把自己的歌排成一份清單」,練團/演出用。

## 3. 技術棧(每個是什麼、為什麼用)

| 分層 | 用什麼 | 白話解釋 |
|---|---|---|
| 前端畫面 | **React + TypeScript + Vite** | 使用者在瀏覽器看到、點的東西。TypeScript = 有型別檢查的 JavaScript,較不易出錯。Vite = 開發伺服器+打包工具。 |
| 前端樣式 | **TailwindCSS** | 用一堆小 class(如 `text-sm`、`px-4`)直接排版,不用另寫 CSS 檔。 |
| 前端狀態 | **Zustand** | 跨元件共用的資料(例如「目前登入者」)放這裡。 |
| 後端 | **Java 21 + Spring Boot** | 提供 API、處理商業邏輯、跟資料庫溝通、驗證登入。 |
| 後端安全 | **Spring Security + JWT** | 判斷「你是誰、能不能做這件事」。 |
| 即時協作 | **WebSocket(STOMP over SockJS)** | 讓伺服器能「主動推」更新給瀏覽器(HTTP 做不到)。 |
| 資料庫 | **PostgreSQL** | 把資料永久存起來的地方。 |
| 資料庫版本 | **Flyway** | 用一支支 SQL 檔(V1、V2…)管理資料表的演進。 |
| 打包執行 | **Docker / Docker Compose** | 把三個服務裝箱、一鍵跑起來。 |
| 測試 | **JUnit(後端)、Vitest(前端)** | 自動驗證程式碼行為正確。 |

## 4. 一個請求怎麼流動(最重要的一張圖)

以「使用者打開『我的歌曲』頁」為例:

```
[瀏覽器 React 頁面]
   MySongsPage 一載入,呼叫 songsApi.listMine()
        │  HTTP GET /api/me/songs   (帶著 JWT token)
        ▼
[nginx / Vite 代理]  把 /api/... 轉給後端
        ▼
[後端 Spring Boot]
   JwtAuthFilter        → 檢查 token,認出「你是誰」
   SongController       → 對應 GET /me/songs 這條路由
   SongService          → 商業邏輯(查我建立的歌、組資料)
   SongRepository       → 下 SQL 查資料庫
        ▼
[PostgreSQL]  回傳資料列
        ▲
   一路組成 JSON:{ "data": { "songs": [...] }, "error": null }
        │
        ▼
[瀏覽器]  React 收到資料,更新畫面,列出歌曲
```

**記住這個順序**:`Controller → Service → Repository → 資料庫`(後端);
`頁面元件 → api 檔 → 後端`(前端)。幾乎每個功能都是這條路。

## 5. 專案資料夾長怎樣(Monorepo)

「Monorepo」= 前端、後端、文件都放在同一個 git 倉庫。
```
ChordCrew/
├── backend/            後端(Java + Spring Boot,用 Gradle 建置)
│   └── src/main/java/com/bandsheet/   ← 後端程式碼都在這
│   └── src/main/resources/db/migration/  ← 資料庫 SQL(V1~V5)
│   └── src/test/       ← 後端測試
├── frontend/           前端(React + Vite)
│   └── src/            ← 前端程式碼都在這
├── docker-compose.yml  一鍵啟動三個服務的設定
├── band-sheet-docs/docs/   需求規格文件(PRD、API_SPEC…)
├── docs/onboarding/    ← 你正在看的新手文件
└── CLAUDE.md           給 AI 助理的專案規則
```

後端 `com/bandsheet/` 底下依「功能」分包:
| 資料夾 | 負責 |
|---|---|
| `auth/` | 註冊、登入、JWT、使用者 |
| `band/` | 樂團、成員、邀請、權限檢查 |
| `song/` | 歌曲 CRUD、分享、最愛(`favorite/`)、和弦邏輯(`chord/`) |
| `sheet/` | 版本歷史 |
| `collab/` | 即時協作(WebSocket、段落鎖) |
| `playlist/` | 歌單 |
| `common/` | 共用工具(統一回傳格式、例外處理、BaseEntity) |
| `config/` | Spring 設定(安全、WebSocket) |

前端 `src/` 底下依「角色」分:
| 資料夾 | 負責 |
|---|---|
| `features/` | 各頁面/功能(songs、playlists、bands、editor、auth…) |
| `api/` | 呼叫後端的函式(每個對應一組後端 API) |
| `stores/` | 全域狀態(Zustand):authStore、collabStore |
| `components/` | 共用元件(AppLayout、Pagination、RequireAuth…) |
| `lib/chord/` | 和弦解析/移調的純邏輯(前後端共用同一套規則) |
| `ws/` | WebSocket 客戶端 |

## 6. 前後端「合約」:統一的回傳格式

**所有後端 API 都回這個形狀**(很重要,前端都依賴它):
```json
{ "data": { ... }, "error": null }
```
成功時 `data` 有值、`error` 為 null;失敗時 `data` 為 null、`error` 有 `{ code, message }`。
這由後端的 `ApiResponse` 類別統一產生,前端就能穩定地讀 `res.data.data.xxx`。
